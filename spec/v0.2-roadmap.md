# v0.2 Roadmap: Closing the Gap with libavoid

This document analyzes the four features needed to make `libavoided-js` a practical standalone edge router for use alongside elk.js and similar layout engines.

## Context

`libavoided-js` is a deterministic orthogonal edge router with obstacle avoidance. The goal is to fill the gap that elk.js cannot fill: standalone edge routing for pre-positioned nodes in pure JavaScript, without WASM or LGPL licensing constraints.

elk.js **does not use libavoid** and **cannot** — it's GWT-transpiled Java, and libavoid is C++. The ELK-libavoid bridge only works on the Java side via a native `libavoid-server` executable. elk.js also cannot accept custom edge routing plugins. This library fills exactly that gap.

### What's already solid

- Zero-dependency pure TypeScript
- MIT license (vs libavoid's LGPL 2.1)
- Deterministic output (same input → same output)
- ELK-compatible output format (sections with startPoint/endPoint/bendPoints)
- Grid-based A* with bend and crossing penalties
- Basic parallel segment spacing and obstacle nudging
- Scaffolding for incremental routing

### What's missing (this roadmap)

| # | Feature | Priority | Rationale |
|---|---------|----------|-----------|
| 1 | Robustness hardening | Highest | Silent fallback through obstacles is a correctness bug |
| 2 | Port-side awareness | High | elk.js outputs ports with PORT_SIDE; current API is too low-level |
| 3 | Nudging overhaul | High | Directly determines visual quality of output |
| 4 | True incremental routing | Medium | Enables interactive use cases beyond batch layout |

## Feature 1: Robustness Hardening

### Problem

When `findPath` returns `null`, `router.ts:89-91` silently falls back to a direct two-point line `[sourcePoint, targetPoint]`, which may cut through obstacles. The consumer has no way to know the route failed. Additionally, `grid.ts` has O(V) linear scans in `hasVertexBetween` that hurt performance on larger graphs.

### Changes

**`types.ts`** — Add `routed: boolean` flag to section output:
```typescript
interface Section {
  startPoint: Point;
  endPoint: Point;
  bendPoints: Point[];
  incomingShape?: string;
  outgoingShape?: string;
  routed: boolean;  // false when pathfinding failed and this is a fallback
}
```

**`router.ts`** — Grid expansion retry:
- When `findPath` returns null, expand the grid with additional scan lines around the source and target, then retry once
- If still null, use fallback and mark `routed: false`

**`grid.ts`** — Performance improvement:
- Index vertices by x-coordinate and y-coordinate using Maps for O(1) lookup in `hasVertexBetween`
- Normalize degenerate inputs: clamp zero/negative obstacle dimensions, skip duplicate obstacles

**`router.ts`** — Input normalization:
- Filter out zero-size obstacles before grid construction
- Deduplicate obstacles at identical positions

### Scope

~200 lines across `router.ts`, `grid.ts`, `types.ts`. ~200 lines of tests. Low risk — each fix is independent and localized.

## Feature 2: Port-Side Awareness

### Problem

elk.js outputs ports with position and side (NORTH/SOUTH/EAST/WEST). The current API requires consumers to manually compute exact `sourcePoint`/`targetPoint` coordinates and `sourceDirection`/`targetDirection` values. This is error-prone and creates unnecessary glue code.

### Changes

**`types.ts`** — Add port specification:
```typescript
type PortSide = 'NORTH' | 'SOUTH' | 'EAST' | 'WEST';

interface PortSpec {
  side: PortSide;
  offset?: number;  // 0-1, position along the side (default: 0.5 = center)
}

interface EdgeRequest {
  // ... existing fields ...
  sourcePort?: PortSpec;
  targetPort?: PortSpec;
}
```

**`router.ts`** — Auto-compute connection points:
- If `sourcePort` is specified, derive `sourcePoint` from the source obstacle's geometry + port spec
- If `targetPort` is specified, derive `targetPoint` from the target obstacle geometry + port spec
- Automatically set `sourceDirection`/`targetDirection` from the port side
- When multiple edges connect to the same side of a node, distribute their attachment points evenly along that side

### Scope

~50 lines in `types.ts`, ~150 lines in `router.ts`. ~150 lines of tests. Low risk — purely additive, backward-compatible.

## Feature 3: Nudging Overhaul

### Problem

The current nudging in `simplify.ts` is the weakest part of the library. `spaceParallelSegments` groups overlapping segments and spreads them, but:
- No ordering awareness: segment assignment to positions is arbitrary (iteration order)
- No channel centering: a single edge through a gap hugs one side instead of centering
- No post-nudge validation: shifted segments may overlap obstacles
- No shared endpoint handling: edges meeting at the same point don't fan out properly

### Changes

**`simplify.ts`** — Rewrite nudging pipeline:

1. **Ordering-aware spacing**: Sort parallel segments by their source/target topology to create visually predictable layouts. Edges going to similar destinations stay together.

2. **Channel centering**: When a single edge passes through a gap between obstacles, center it in the available space rather than leaving it at the grid coordinate.

3. **Post-nudge validation**: After all nudging, verify no segment now overlaps an expanded obstacle. If it does, clamp it to the nearest valid position.

4. **Shared endpoint fan-out**: When multiple edges share an endpoint, offset their connection points slightly to avoid visual overlap at the junction.

**New `RouterOptions` fields:**
```typescript
interface RouterOptions {
  // ... existing fields ...
  nudgeConnectedSegments?: boolean;  // nudge apart segments touching shapes (default: true)
  centerInChannels?: boolean;        // center single edges in gaps (default: true)
}
```

### Scope

~350 lines rewritten/added in `simplify.ts`. ~200 lines of tests. Low risk — isolated to simplification pass, doesn't affect grid or pathfinding.

## Feature 4: True Incremental Routing

### Problem

`incremental.ts` has good change detection (`edgeTopologyChanged`) but when anything changes, it falls through to `routeEdges(fullInput, options)` on line 126 — full grid rebuild, full re-route of all edges. This defeats the purpose of incremental routing for interactive use cases.

### Changes

**`grid.ts`** — Grid mutation support:
- Add `addObstacleToGrid(grid, obstacle, margin)` — insert new scan lines and reconnect vertices
- Add `removeObstacleFromGrid(grid, obstacleId, margin)` — remove scan lines that are no longer needed
- Add `updateObstacleInGrid(grid, oldObstacle, newObstacle, margin)` — combined remove + add

**`incremental.ts`** — Selective re-routing:
- Maintain a persistent `Grid` across calls (store it in the output or accept it as a parameter)
- When obstacles change: mutate the grid incrementally, then identify which edges are affected (their path passes within margin of a changed obstacle)
- Re-route only affected edges, preserving the `RoutedSegments` from unchanged edges
- When only edge endpoints change: keep the grid, re-route just those edges

**`types.ts`** — Extend output for state persistence:
```typescript
interface RouterOutput {
  edges: RoutedEdge[];
  _state?: RouterState;  // opaque state for incremental reuse
}
```

### Scope

~300 lines in `grid.ts` for grid mutation, ~150 lines rewriting `incremental.ts`, ~50 lines adapting `router.ts`. ~250 lines of tests. Medium risk — grid mutation is the hardest part and needs careful validation that the graph stays consistent.

## Implementation Order

```
1. Robustness hardening  (no dependencies, fixes correctness bugs)
2. Port-side awareness   (builds on robust router)
3. Nudging overhaul      (builds on robust router, port-aware endpoints)
4. Incremental routing   (builds on all of the above)
```

Each feature is independently shippable and testable. The natural ordering ensures each step benefits from the previous one being solid.

## Estimated Total Scope

| Feature | Code | Tests | Risk |
|---------|------|-------|------|
| Robustness hardening | ~200 lines | ~200 lines | Low |
| Port-side awareness | ~200 lines | ~150 lines | Low |
| Nudging overhaul | ~350 lines | ~200 lines | Low |
| Incremental routing | ~500 lines | ~250 lines | Medium |
| **Total** | **~1,250 lines** | **~800 lines** | |

This roughly doubles the library's sophistication without doubling its size. The existing ~1,000 lines of source and ~1,900 lines of tests provide a solid foundation.

## Non-Goals for v0.2

- **Polyline routing**: Requires a visibility graph (different algorithm entirely). Orthogonal is what elk.js consumers need.
- **Dynamic callbacks**: elk.js is batch-oriented. Trivial to add later when an interactive consumer asks for it.
- **Cluster support**: Would require ClusterRef-style boundaries. Valuable but not needed for the primary elk.js use case.
- **Hyperedge routing**: JunctionRef-style multi-target connectors. Significant complexity for a niche feature.
